# 功能模块设计

## 1. 项目定位
- 目标：将当前项目改造成面向校园大型活动的报名与签到平台。
- 第一阶段范围：活动门户、活动详情、活动发起与编辑、报名、我的报名、报名名单、凭证化签到核销。
- 访问约束：除 `login.html`、`login2.html` 与登录相关接口外，整站页面和业务接口默认要求先登录。
- 登录态策略：前端 token 持久化到 `localStorage`，浏览器关闭后只要 token 未过期仍保持登录。
- 当前权限模型：
  - 鉴权模型升级为正式 RBAC：`用户 -> 角色 -> 权限`
  - 兼容字段：`tb_user.role_type` 暂时保留，只用于兼容旧登录态与历史数据迁移
- 当前阶段角色：
    - `USER`：浏览活动、报名、查看个人报名与签到状态
    - `ACTIVITY_ADMIN`：发起活动、编辑自己创建的活动、查看自己活动的报名名单与签到配置
    - `PLATFORM_ADMIN`：预留平台治理能力，第一阶段不开放独立前台入口
  - 默认注册策略：
    - 所有新注册用户默认仅绑定 `USER`
    - 普通用户需要提交“成为活动主办方”申请，经平台管理员审核通过后追加绑定 `ACTIVITY_ADMIN`
    - 平台管理员采用系统预置账号：`admin / 123456`
  - 数据范围约束：
    - 活动管理员即使具备管理权限，也只能操作自己创建的活动
    - 平台级 RBAC 只负责“是否能访问该类接口”，具体是否能操作该活动仍受 `creator_id` 等数据权限限制

## 2. 模块划分

### 2.1 活动门户模块
- 页面入口：`front/html/campus/index.html`
- 后端接口：`GET /activity/public/list`、`GET /activity/public/categories`
- 主要能力：
  - 按关键字查询活动
  - 按分类筛选活动
  - 支持按活动地点、主办方、活动开始时间范围补充筛选
  - 支持综合排序、即将开始、最新发布、报名人数最多、热门优先
  - 查看报名状态、人数和活动时间
  - 主办方账号在首页显示“发布活动”入口，直接进入活动管理页

### 2.2 活动详情模块
- 页面入口：`front/html/campus/activity-detail.html`
- 后端接口：`GET /activity/public/{id}`
- 主要能力：
  - 展示活动摘要、详情、时间、地点、主办方
  - 学生用户已报名后查看自己的唯一签到凭证号与签到状态
  - 活动主办方查看签到凭证核销面板、签到统计与最近核销记录
  - 学生用户未报名时发起报名
  - 页面采用蓝灰色卡片风格，与首页和发布页统一视觉语言

### 2.3 活动管理模块
- 页面入口：`front/html/campus/activity-manage.html`
- 后端接口：`POST /activity`、`PUT /activity`、`GET /activity/manage/mine`
- 主要能力：
  - 创建活动
  - 编辑活动
  - 上传多张活动图片到阿里云 OSS，默认首图作为活动封面
  - 编辑页采用类似朋友圈发布的单列图文流排版，优先突出标题、正文、图片和时间信息
  - 管理页保持纯编辑态，不在编辑页面展示活动概览和“我发起的活动”侧栏
  - 热点活动详情与列表走 Redis 缓存，降低数据库读压力
  - 创建/编辑活动后同步刷新活动缓存、报名元数据与剩余名额
  - 创建、编辑、报名确认、退出报名后通过 RocketMQ 异步同步 Elasticsearch 索引
  - 仅主办方账号显示“发起”活动入口和活动管理页
  - 主办方新建或编辑活动后默认进入“待平台审核”状态，审核通过后才会出现在公开活动门户

#### 2.3.1 高并发设计
- 缓存目标：
  - 活动详情缓存：`cache:activity:detail:{activityId}`
  - 活动列表缓存：`cache:activity:list:{version}:{queryHash}`
  - 活动分类缓存：`cache:activity:categories:{version}`
- 报名并发控制：
  - Redis Hash：`activity:meta:{activityId}` 保存活动状态、报名窗口、人数上限
  - Redis String：`activity:slots:{activityId}` 保存剩余名额
  - Redis Set：`activity:register:users:{activityId}` 保存已报名用户，防止重复报名
  - RocketMQ Topic：`activity-register-topic` 异步削峰，报名请求通过 Lua 原子校验后投递消息
  - 一致性策略：
  - Redis Lua 脚本原子完成“校验活动状态、校验报名时间、校验重复报名、冻结名额、写入用户待确认状态”
  - RocketMQ 消费者异步确认 `tb_activity_registration` 并条件更新 `tb_activity.registered_count`
  - MySQL 保留 `activity_id + user_id` 唯一约束，保证最终不会超额或重复报名
  - 落库失败或消息发送失败时执行 Redis 补偿：回补剩余名额、释放冻结名额并写入失败状态

#### 2.3.2 搜索引擎升级设计
- 检索架构：
  - MySQL 仍是活动主数据源
  - Elasticsearch 负责活动大厅关键词搜索、条件筛选、排序和分类聚合
  - 公共列表接口优先走 ES，查询失败自动降级到 MySQL + Redis 缓存
- 索引名称：
  - 默认索引：`activity_index_dev`
- 索引文档字段：
  - `title`、`summary`、`content`、`organizerName`、`location`：全文检索字段
  - `category`、`status`、`stageStatus`：精确过滤与聚合字段
  - `registrationStartTime`、`registrationEndTime`、`eventStartTime`、`eventEndTime`、`createTime`、`updateTime`：时间过滤与排序字段
  - `registeredCount`、`maxParticipants`、`heatScore`、`isHot`：热度与排序字段
- 查询结构：
  - `must`：关键词在标题、简介、主办方、地点、正文做多字段匹配，标题权重更高
  - `filter`：分类、状态、活动开始时间范围、主办方、地点
  - `sort`：综合排序、开始时间升序、发布时间倒序、报名人数倒序、热度倒序
- 同步链路：
  - 活动创建/编辑、报名成功、退出报名后，业务服务在事务提交后投递 `activity-search-sync-topic`
  - 搜索消费者按活动 ID 增量更新或删除 ES 文档
  - 定时任务按 `update_time` 增量修复索引，兜底 MQ 丢失或同步失败场景
  - 应用启动时若检测到 `activity_index_dev` 已存在但分片不可用，会自动删除坏索引并按 MySQL 已发布活动全量重建
- 页面入口：
  - `front/html/campus/index.html`
- 相关容器：
  - `docker-compose.yml` 新增 `elasticsearch` 单节点服务

### 2.4 报名模块
- 页面入口：`front/html/campus/activity-detail.html`、`front/html/campus/info.html`
- 后端接口：`POST /activity/{id}/register`、`GET /activity/registration/mine`
- 主要能力：
  - 在报名时间窗口内报名，先返回 `PENDING_CONFIRM`
  - 活动开始前允许已报名用户退出活动
  - 限制重复报名
  - 控制人数上限
  - 查看个人报名记录
  - 通过 `GET /activity/{id}/register/status` 查询最终报名状态
  - 热点活动通过 Redis + Lua + RocketMQ 保证高并发下不超出报名上限
  - WebSocket 事件 `activity_register_result` 主动推送最终结果

### 2.5 签到模块
- 页面入口：`front/html/campus/activity-detail.html`、`front/html/campus/activity-manage.html`
- 后端接口：`POST /activity/manage/{id}/check-in/verify`、`GET /activity/manage/{id}/check-in/stats`、`GET /activity/manage/{id}/check-in/records`
- 主要能力：
  - 用户报名成功后自动生成签到凭证
  - 学生仅可查看自己的签到凭证
  - 工作人员输入展示码或凭证 ID 完成核销签到
  - 限制重复签到并记录核销人、签到时间
  - 统计已签到人数、未签到人数和签到率

### 2.6 用户资料完善模块
- 页面入口：
  - 个人中心：`front/html/campus/info.html`（新增“编辑资料”入口）
  - 资料编辑：`front/html/campus/info-edit.html`（单页表单：头像上传 + 资料保存）
- 后端接口：
  - `GET /user/me`（获取当前登录用户基础信息）
  - `GET /user/info/{id}`（获取用户资料详情）
  - `PUT /user/profile`（更新当前用户资料：学院/年级/导师等）
  - `POST /user/avatar`（上传头像到 OSS，并更新当前用户头像 URL）
- 主要能力：
  - 登录用户完善个人资料：学院、年级、导师、个人介绍、性别、城市、生日
  - 上传并更新头像（存储到阿里云 OSS，前端直接使用 URL 展示）
  - 更新昵称与头像后同步刷新登录态缓存，页面无需重新登录即可看到变化
  - 编辑页采用与活动门户一致的蓝灰色卡片式布局

### 2.8 主办方申请模块
- 页面入口：
  - 个人中心：`front/html/campus/info.html`
- 后端接口：
  - `POST /user/organizer/apply`
  - `GET /user/organizer/apply/me`
- 主要能力：
  - 普通用户在个人中心提交主办方申请
  - 提交项至少包含组织名称与申请理由
  - 已申请用户可查看最近一次申请状态：`PENDING`、`APPROVED`、`REJECTED`
  - 审核通过后，系统为该用户追加 `ACTIVITY_ADMIN` 角色

### 2.9 平台管理员后台模块
- 页面入口：
  - 管理后台：`front/html/campus/admin-dashboard.html`
- 后端接口：
  - `GET /activity/admin/review-list`
  - `POST /activity/admin/{id}/review`
  - `GET /user/admin/organizer-applications`
  - `POST /user/admin/organizer-applications/{id}/review`
- 主要能力：
  - 管理员登录后直接进入后台
  - 审核主办方创建的待发布活动
  - 审核普通用户成为活动主办方的申请
  - 管理后台仅 `PLATFORM_ADMIN` 可访问

## 2.7 首页头像入口
- 页面入口：
  - 活动门户：`front/html/campus/index.html`
- 主要能力：
  - 首页头图区域右上角显示头像入口
  - 已登录点击进入个人信息页 `info.html`
  - 未登录点击进入登录页 `login.html`
  - 头像入口只保留在首页，其它页面不再悬浮显示
  - 个人页提供返回首页按钮，其它业务页保留页面内返回按钮

## 5. 约束与默认策略
- 活动公开浏览不需要登录。
- 活动创建、编辑、报名、签到、报名名单查看必须登录。
- 具备 `activity:create` 权限的账号显示“发起”活动入口与“我发起的活动”页签；无该权限账号隐藏这些模块。
- 具备 `registration:create` 权限的账号才允许报名；具备 `registration:view_self` 才允许查看“我的报名”。
- 普通用户默认只能注册为 `USER`，不能自行直接成为主办方。
- 活动主办方资格必须经过平台管理员审核。
- 主办方创建或编辑活动后默认进入待审核状态，未审核活动不出现在活动门户。
- 学生用户在活动开始前可退出已报名活动；退出后释放名额并删除签到凭证。
- 所有页面移除底部导航栏。
- 仅首页保留头像入口，其它页面依赖页面内按钮完成返回和跳转。
- 当前版本对活动创建、编辑、报名名单、签到核销、签到记录等关键接口增加后端权限校验。
- 第一阶段先完成 RBAC 基础设施和关键接口收口，平台管理员入口与角色管理页面后续补齐。
- 当前版本已补充平台管理员后台页面，用于活动审核与主办方申请审核。
- 第一阶段签到方式固定为“报名签发凭证 + 工作人员核销”。
- 学生只能查看自己的凭证，不允许学生直接调用签到核销接口。
- 公开活动详情不会暴露所有签到凭证，只向已报名用户返回自己的凭证摘要。
- 用户头像采用 OSS 存储与公网 URL 展示；资料编辑必须登录。
- 活动图片采用 OSS 存储；`images` 保存多图 URL，`cover_image` 默认取首图。
- 热点活动报名采用“Redis 冻结名额 + RocketMQ 异步确认 + WebSocket 推送 + 失败补偿”策略。
- 活动详情、列表、分类默认启用 Redis 缓存；创建、编辑、报名签发凭证与签到核销后触发缓存失效或重建。
- 活动搜索默认启用 Elasticsearch；当 ES 不可用时，活动大厅降级回 MySQL 列表查询，不影响主链路可用性。
- 第一阶段前端统一采用蓝灰色、圆角卡片、轻阴影的界面风格，避免页面视觉割裂。
- 登录后 token 缓存同时保存 `roleCodes` 与 `permissions`，请求进入后恢复到当前登录态，用于后端接口鉴权。
