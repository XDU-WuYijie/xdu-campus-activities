# 功能模块设计

## 1. 项目定位
- 目标：将当前项目改造成面向校园大型活动的报名与签到平台。
- 第一阶段范围：活动门户、活动详情、活动发起与编辑、报名、我的报名、报名名单、凭证化签到核销。
- 当前角色模型：
  - 主办方账号：可发起活动、编辑自己创建的活动、查看自己活动的报名名单与签到配置
  - 学生账号：仅可浏览、报名和签到，看不到创建活动的模块

## 2. 模块划分

### 2.1 活动门户模块
- 页面入口：`front/html/campus/index.html`
- 后端接口：`GET /activity/public/list`、`GET /activity/public/categories`
- 主要能力：
  - 按关键字查询活动
  - 按分类筛选活动
  - 查看报名状态、人数和活动时间

### 2.2 活动详情模块
- 页面入口：`front/html/campus/activity-detail.html`
- 后端接口：`GET /activity/public/{id}`
- 主要能力：
  - 展示活动摘要、详情、时间、地点、主办方
  - 学生用户已报名后查看自己的唯一签到凭证号与签到状态
  - 活动主办方查看签到凭证核销面板、签到统计与最近核销记录
  - 学生用户未报名时发起报名
  - 页面采用蓝灰色卡片风格，与首页和发布页统一视觉语言

### 2.3 活动管理模块
- 页面入口：`front/html/campus/activity-manage.html`
- 后端接口：`POST /activity`、`PUT /activity`、`GET /activity/manage/mine`
- 主要能力：
  - 创建活动
  - 编辑活动
  - 上传多张活动图片到阿里云 OSS，默认首图作为活动封面
  - 查看自己发起的活动
  - 工作人员按凭证 ID 或展示码核销签到
  - 查看签到统计与核销记录
  - 查看报名名单
  - 编辑页视觉风格与资料编辑页保持一致，采用简洁蓝灰卡片布局
  - 热点活动详情与列表走 Redis 缓存，降低数据库读压力
  - 创建/编辑活动后同步刷新活动缓存、报名元数据与剩余名额
  - 仅主办方账号显示“发起”活动入口和活动管理页

#### 2.3.1 高并发设计
- 缓存目标：
  - 活动详情缓存：`cache:activity:detail:{activityId}`
  - 活动列表缓存：`cache:activity:list:{version}:{queryHash}`
  - 活动分类缓存：`cache:activity:categories:{version}`
- 报名并发控制：
  - Redis Hash：`activity:meta:{activityId}` 保存活动状态、报名窗口、人数上限
  - Redis String：`activity:slots:{activityId}` 保存剩余名额
  - Redis Set：`activity:register:users:{activityId}` 保存已报名用户，防止重复报名
  - RocketMQ Topic：`activity-register-topic` 异步削峰，报名请求通过 Lua 原子校验后投递消息
  - 一致性策略：
  - Redis Lua 脚本原子完成“校验活动状态、校验报名时间、校验重复报名、冻结名额、写入用户待确认状态”
  - RocketMQ 消费者异步确认 `tb_activity_registration` 并条件更新 `tb_activity.registered_count`
  - MySQL 保留 `activity_id + user_id` 唯一约束，保证最终不会超额或重复报名
  - 落库失败或消息发送失败时执行 Redis 补偿：回补剩余名额、释放冻结名额并写入失败状态

### 2.4 报名模块
- 页面入口：`front/html/campus/activity-detail.html`、`front/html/campus/info.html`
- 后端接口：`POST /activity/{id}/register`、`GET /activity/registration/mine`
- 主要能力：
  - 在报名时间窗口内报名，先返回 `PENDING_CONFIRM`
  - 活动开始前允许已报名用户退出活动
  - 限制重复报名
  - 控制人数上限
  - 查看个人报名记录
  - 通过 `GET /activity/{id}/register/status` 查询最终报名状态
  - 热点活动通过 Redis + Lua + RocketMQ 保证高并发下不超出报名上限
  - WebSocket 事件 `activity_register_result` 主动推送最终结果

### 2.5 签到模块
- 页面入口：`front/html/campus/activity-detail.html`、`front/html/campus/activity-manage.html`
- 后端接口：`POST /activity/manage/{id}/check-in/verify`、`GET /activity/manage/{id}/check-in/stats`、`GET /activity/manage/{id}/check-in/records`
- 主要能力：
  - 用户报名成功后自动生成签到凭证
  - 学生仅可查看自己的签到凭证
  - 工作人员输入展示码或凭证 ID 完成核销签到
  - 限制重复签到并记录核销人、签到时间
  - 统计已签到人数、未签到人数和签到率

### 2.6 用户资料完善模块
- 页面入口：
  - 个人中心：`front/html/campus/info.html`（新增“编辑资料”入口）
  - 资料编辑：`front/html/campus/info-edit.html`（单页表单：头像上传 + 资料保存）
- 后端接口：
  - `GET /user/me`（获取当前登录用户基础信息）
  - `GET /user/info/{id}`（获取用户资料详情）
  - `PUT /user/profile`（更新当前用户资料：学院/年级/导师等）
  - `POST /user/avatar`（上传头像到 OSS，并更新当前用户头像 URL）
- 主要能力：
  - 登录用户完善个人资料：学院、年级、导师、个人介绍、性别、城市、生日
  - 上传并更新头像（存储到阿里云 OSS，前端直接使用 URL 展示）
  - 更新昵称与头像后同步刷新登录态缓存，页面无需重新登录即可看到变化
  - 编辑页采用与活动门户一致的蓝灰色卡片式布局

## 2.7 首页头像入口
- 页面入口：
  - 活动门户：`front/html/campus/index.html`
- 主要能力：
  - 首页头图区域右上角显示头像入口
  - 已登录点击进入个人信息页 `info.html`
  - 未登录点击进入登录页 `login.html`
  - 头像入口只保留在首页，其它页面不再悬浮显示
  - 个人页提供返回首页按钮，其它业务页保留页面内返回按钮

## 3. 数据模型

### 3.1 活动表 `tb_activity`
- 核心字段：
  - `creator_id`
  - `organizer_name`
  - `title`
  - `cover_image`
  - `images`
  - `summary`
  - `content`
  - `category`
  - `location`
  - `max_participants`
  - `registered_count`
  - `registration_start_time`
  - `registration_end_time`
  - `event_start_time`
  - `event_end_time`
  - `check_in_enabled`
  - `check_in_code`
  - `check_in_code_expire_time`
  - `status`

### 3.2 报名表 `tb_activity_registration`
- 核心字段：
  - `activity_id`
  - `user_id`
  - `status`
  - `request_id`
  - `fail_reason`
  - `voucher_id`
  - `check_in_status`
  - `check_in_time`
  - `confirm_time`
- 约束：
  - `activity_id + user_id` 唯一，防止重复报名

### 3.3 签到凭证表 `tb_activity_voucher`
- 核心字段：
  - `id`
  - `display_code`
  - `activity_id`
  - `registration_id`
  - `user_id`
  - `status`
  - `issued_time`
  - `checked_in_time`
  - `checked_in_by`

### 3.4 签到核销记录表 `tb_activity_check_in_record`
- 核心字段：
  - `activity_id`
  - `voucher_id`
  - `user_id`
  - `operator_id`
  - `result_status`
  - `request_key`

### 3.3 用户表与资料表
- 用户表 `tb_user`
  - 核心字段：
    - `nick_name` 用户昵称
    - `icon` 用户头像 URL（OSS 公网 URL）
    - `role_type` 账号类型：`1` 主办方，`2` 学生
- 用户资料表 `tb_user_info`
  - 核心字段（新增）：
    - `college` 所属学院
    - `grade` 所属年级
    - `mentor` 导师

## 4. 接口设计
- 公开接口：
  - `GET /activity/public/list`
  - `GET /activity/public/categories`
  - `GET /activity/public/{id}`
- 登录后接口：
  - `POST /activity`
  - `PUT /activity`
  - `GET /activity/manage/mine`
  - `POST /activity/manage/image`
  - `POST /activity/{id}/register`
  - `GET /activity/{id}/register/status`
  - `DELETE /activity/{id}/register`
  - `GET /activity/registration/mine`
  - `GET /activity/manage/{id}/registrations`
  - `POST /activity/manage/{id}/check-in/verify`
  - `GET /activity/manage/{id}/check-in/stats`
  - `GET /activity/manage/{id}/check-in/records`
  - `PUT /user/profile`
  - `POST /user/avatar`
  - `GET /user/me` 返回当前登录用户角色，用于前端权限显隐

## 4.1 内部缓存与异步资源
- Lua 脚本：`src/main/resources/activity_register.lua`
- RocketMQ Topic：`activity-register-topic`
- WebSocket：`/ws/activity-registration?token={token}`
- 缓存版本 Key：`cache:activity:version`

## 5. 约束与默认策略
- 活动公开浏览不需要登录。
- 活动创建、编辑、报名、签到、报名名单查看必须登录。
- 主办方账号显示“发起”活动入口与“我发起的活动”页签；学生账号隐藏这些模块。
- 主办方账号不能报名参加任何活动，活动详情页不显示报名入口。
- 学生用户在活动开始前可退出已报名活动；退出后释放名额并删除签到凭证。
- 所有页面移除底部导航栏。
- 仅首页保留头像入口，其它页面依赖页面内按钮完成返回和跳转。
- 当前版本仅做前端权限隐藏，不对活动创建/编辑/管理接口增加后端强校验。
- 第一阶段不引入独立平台管理员角色。
- 第一阶段签到方式固定为“报名签发凭证 + 工作人员核销”。
- 学生只能查看自己的凭证，不允许学生直接调用签到核销接口。
- 公开活动详情不会暴露所有签到凭证，只向已报名用户返回自己的凭证摘要。
- 用户头像采用 OSS 存储与公网 URL 展示；资料编辑必须登录。
- 活动图片采用 OSS 存储；`images` 保存多图 URL，`cover_image` 默认取首图。
- 热点活动报名采用“Redis 冻结名额 + RocketMQ 异步确认 + WebSocket 推送 + 失败补偿”策略。
- 活动详情、列表、分类默认启用 Redis 缓存；创建、编辑、报名签发凭证与签到核销后触发缓存失效或重建。
- 第一阶段前端统一采用蓝灰色、圆角卡片、轻阴影的界面风格，避免页面视觉割裂。
