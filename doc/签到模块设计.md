content = """# 校园活动平台用户签到功能设计方案

## 1. 功能定位

在校园活动场景中，用户报名成功只是第一步，真正落地还需要解决“到场核验”的问题。  
例如校园歌手大赛、中秋游园会、名师讲座、社团活动等场景，主办方通常会遇到以下问题：

- 现场靠纸质名单核对，效率低
- 用户可能重复签到
- 工作人员难以快速判断用户是否真的报名成功
- 活动结束后不好统计真实到场人数和签到率
- 热门活动在签到高峰期容易出现重复请求和状态错乱

因此，签到模块不是一个简单的“点击签到”功能，而是活动履约闭环的重要一环。它的目标包括：

1. 让报名成功的用户拥有一个可核验的活动凭证  
2. 让工作人员在现场快速完成签到  
3. 保证同一张凭证只能成功签到一次  
4. 让主办方可以统计签到率、到场人数和签到名单

---

## 2. 整体设计思路

本模块不采用二维码方案，而采用**内部凭证 ID + 展示码**的核验方式。

### 2.1 凭证生成
用户报名成功后，系统不会只记录一条报名记录，而是会进一步签发一张“活动凭证”。

这张凭证采用两层标识：

- **内部凭证 ID**：供系统内部使用，保证全局唯一，便于后端查库、状态流转和问题追踪
- **展示码**：供前端展示和工作人员现场手工输入使用，长度更短，便于人工核验

这样设计的原因是：

- 内部 ID 适合系统内部管理
- 展示码更短，更适合现场人工输入
- 不会直接把过长的系统内部主键暴露给前端

也就是说，用户报名成功后，系统会生成一张“有效凭证”，后续签到流程全部围绕这张凭证进行。

---

## 3. 签到主流程设计

### 3.1 工作人员发起签到
签到时不是让用户自己点击“我已签到”，而是由工作人员端发起签到操作。

工作人员可以通过以下两种方式进行核验：

- 输入展示码
- 输入内部凭证 ID

然后前端调用签到接口，把 `activityId` 和凭证信息传给后端。

---

### 3.2 查询凭证
后端先根据 `voucherId` 或 `displayCode` 查询凭证记录。

如果查不到，直接返回：

- 凭证不存在
- 或凭证无效

---

### 3.3 校验活动是否匹配
因为一个用户可能参加多个活动，所以不能只验证“凭证是否存在”，还必须验证这张凭证是否属于当前活动。

也就是说：

- 接口中的 `activityId` 必须和凭证绑定的活动一致
- 如果不一致，则不能签到

这样可以避免：

- A 活动的凭证拿到 B 活动签到
- 工作人员误操作导致串场签到

---

### 3.4 校验凭证状态
凭证状态建议设计为：

- `UNUSED`：未使用
- `CHECKED_IN`：已签到
- `CANCELED`：已取消
- `EXPIRED`：已失效

只有 `UNUSED` 状态的凭证才允许继续签到。

如果已经签到过，则直接返回“已签到”；  
如果已取消或已失效，也不能继续签到。

---

### 3.5 校验签到时间窗口
为了防止活动开始前很久或活动结束后仍然能够签到，系统需要增加签到时间窗口校验。

例如可以配置为：

- 活动开始前 30 分钟允许签到
- 活动结束后 30 分钟关闭签到

这样可以防止非活动时间误签到。

---

### 3.6 执行签到更新
在前面的业务校验全部通过后，系统才真正执行签到更新。

这里不能只做“先查未签到，再 update”，因为在并发场景下可能会出现问题。  
更稳妥的方案是使用**条件更新**：

只有当凭证状态是 `UNUSED` 时，才能更新成 `CHECKED_IN`。

逻辑上可以理解为：

- 条件：`voucher_id = ? and status = 'UNUSED'`
- 更新：`status = 'CHECKED_IN'`

如果更新成功，则说明本次签到成功；  
如果更新失败，则说明该凭证已经不处于可签到状态，需要进一步判断是已签到、已取消还是已失效。

---

### 3.7 记录签到结果
签到成功后，系统需要记录以下信息：

- 签到时间
- 操作人（工作人员 ID）
- 活动 ID
- 用户 ID
- 签到状态

这样后续可以做统计、追溯和导出名单。

## 4. 核心技术设计

### 4.1 凭证建模
签到模块的核心不是一个按钮，而是一张“活动凭证”。

建议凭证对象至少包含：

- 内部凭证 ID
- 展示码
- 活动 ID
- 用户 ID
- 凭证状态
- 签发时间
- 签到时间
- 签到操作人

#### 4.1.1 凭证 ID 设计
内部凭证 ID 可以使用：

- UUIDv7
- 或业务唯一 ID 生成方案

展示码则建议单独生成短码，例如 8 位大写字母与数字组合，并去掉容易混淆的字符。

这样做的好处是：

- 内部 ID 保证唯一性和可追踪性
- 展示码便于人工输入
- 前后端职责划分清晰

---

### 4.2 幂等设计
签到功能最容易出问题的地方就是重复请求。  
常见场景包括：

- 工作人员点了两次按钮
- 网络卡顿导致前端自动重试
- 两个工作人员同时帮同一个用户签到
- 页面响应慢，误以为没成功又重复提交

因此，签到模块的幂等设计分为两层：

#### 4.2.1 请求级幂等
前端调用签到接口时，携带一个 `Idempotency-Key`。

后端收到请求后，先去 Redis 检查这个 key 是否已经处理过。

如果没有处理过，就在 Redis 中写入一条记录，状态记为 `PROCESSING`，表示当前请求正在处理。

如果已经处理过，则分情况处理：

- 如果参数和第一次请求一致，且状态已经成功，则直接返回第一次的处理结果
- 如果参数不同，但仍使用同一个幂等 key，则返回冲突错误
- 如果状态仍在处理中，则返回“请求处理中”或提示稍后重试

这一层主要解决：

- 前端重试
- 重复点击
- 短时间重复请求反复打到数据库的问题

---

#### 4.2.2 业务级幂等
仅依赖 Redis 做幂等还不够，因为 Redis 解决的是“请求防重”，数据库才是最终状态来源。

所以数据库更新时，还需要做条件约束：

- 只有状态为 `UNUSED` 的凭证，才允许更新为 `CHECKED_IN`

如果更新影响行数为 1，说明本次签到成功。  
如果更新影响行数为 0，则再查询当前状态并返回明确结果：

- 已签到
- 已取消
- 已失效
- 不存在
- 活动不匹配

这样就保证了：  
即使两个并发请求同时进入，最终也只有一个请求能真正完成签到。

## 5. 为什么不使用二维码

当前版本没有使用二维码，主要是为了先把**核心签到链路**做稳定。

即使不用二维码，签到仍然可以完成，只是凭证载体变成：

- 内部凭证 ID
- 展示码

工作人员通过展示码或凭证 ID 手工输入即可完成核验。

这样设计的优点是：

1. 先把业务闭环跑通，避免一开始把重点放在扫码交互上  
2. 面试时更容易把重点放在凭证建模、状态设计和幂等控制上  
3. 后续如果要升级为二维码，只是前端输入方式发生变化，后端核验逻辑基本不需要推翻

因此可以表述为：

> 当前版本先采用凭证号核验方案，后续可以平滑升级为二维码或动态码方案。

## 6. 主要问题与解决方案

### 6.1 重复签到问题
**问题：**  
工作人员连续点击、网络重试、多端并发操作，都可能导致重复签到。

**解决方案：**
- 使用 `Idempotency-Key + Redis` 做请求级防重
- 使用数据库条件更新做业务级防重

**最终效果：**
- 重复请求最多返回“已签到”
- 不会导致签到状态错乱

---

### 6.2 同一用户参加多个活动的问题
**问题：**  
不能只按用户 ID 做签到，否则会把不同活动混淆。

**解决方案：**
- 必须基于“活动 + 凭证”进行核验
- 校验 `activityId` 和凭证绑定活动是否一致

**最终效果：**
- 避免不同活动签到串场
- 提高现场核验准确性

---

### 6.3 凭证被冒用的问题
**问题：**  
如果凭证码被他人获取，可能存在冒用风险。

**解决方案：**
当前版本主要依赖以下校验降低风险：

- 活动 ID 校验
- 凭证状态校验
- 签到时间窗口校验
- 已签到后不可重复使用

后续若要进一步增强安全性，可以升级为：

- 动态码
- 带签名 token
- 结合账号身份做二次校验

---

### 6.4 网络卡顿导致重复请求的问题
**问题：**  
活动现场网络不稳定时，前端可能自动重试，造成多次重复请求。

**解决方案：**
- 前端请求必须携带 `Idempotency-Key`
- 服务端通过 Redis 记录第一次请求结果
- 后续重复请求直接返回第一次结果，而不是再次执行业务逻辑

**最终效果：**
- 避免网络重试带来的重复签到
- 提升现场系统稳定性

---

### 6.5 两个工作人员同时帮同一用户签到的问题
**问题：**  
在热门活动签到高峰期，可能出现多个工作人员对同一凭证同时操作。

**解决方案：**
- 请求层用 Redis 做第一层防重
- 数据库层使用 `status = UNUSED` 的条件更新做最终兜底

**最终效果：**
- 同一张凭证只会有一个请求签到成功
- 其余请求会被识别为“已签到”

---

### 6.6 主办方如何做签到统计的问题
**问题：**  
活动结束后，主办方需要知道真实到场人数、签到率、未签到名单等信息。

**解决方案：**
签到成功后保留签到记录，并支持统计以下指标：

- 报名人数
- 已签到人数
- 签到率
- 未签到名单
- 签到时间分布

**最终效果：**
- 主办方可以评估活动效果
- 平台可以形成活动履约闭环

## 7. 技术栈建议

这个模块不需要过度复杂，下面这套技术栈就比较稳：

- **Spring Boot**：承载签到服务逻辑
- **Spring MVC**：提供签到相关 API
- **MyBatis / MyBatis-Plus**：操作报名、凭证、签到记录表
- **MySQL**：存储凭证数据和签到结果
- **Redis**：做幂等控制、短期状态缓存、防重复请求
- **JWT**：给工作人员端和管理端做登录鉴权

### 可选增强
- **RabbitMQ**：签到成功后异步通知或异步更新统计数据
- **MinIO**：如果后续要做电子凭证文件存储，可以扩展接入
- **Sentinel**：若活动签到流量较高，可对签到接口进行限流保护

## 9. 模块亮点总结

该签到模块最值得强调的亮点包括：

1. **凭证化设计**：不是简单“按钮签到”，而是围绕活动凭证建立核验机制  
2. **多维校验**：活动匹配、时间窗口、状态判断，保证业务正确性  
3. **双层幂等**：请求级幂等 + 业务级幂等，解决重复签到问题  
4. **统计闭环**：签到完成后支持到场人数、签到率和名单统计  
5. **可扩展性**：当前先用凭证号方案，后续可平滑升级成二维码或动态码方案

## 10. 总结

用户签到功能的核心并不是“提供一个签到接口”，而是围绕“活动履约”设计一套可靠的核验机制。  
它既要保证业务正确，也要能应对现场高频操作、网络抖动和重复请求带来的问题。

因此，本方案的重点不是扫码形式本身，而是：

- 凭证建模
- 状态设计
- 幂等控制
- 签到记录沉淀
- 活动数据闭环
