# 校园活动平台用户报名模块设计方案（Redis 预扣减 + MQ 异步确认 + WebSocket 状态推送）

## 1. 方案概述

本方案面向校园活动报名场景，重点解决热门活动开放报名时的高并发冲击、名额控制、重复报名以及前端状态一致性问题。

与传统“同步插库后直接返回报名成功”的方案不同，本方案采用：

- **Redis 预扣减（冻结名额）**
- **MQ 异步落库**
- **消费者完成最终确认**
- **WebSocket 主动推送报名结果**

用户在点击报名后，系统不会立即返回“报名成功”，而是先返回“**报名确认中**”。  
等到消息消费者真正完成数据库落库、状态确认后，再通过 WebSocket 通知前端，将页面状态更新为：

- 报名成功
- 报名失败
- 名额不足
- 重复报名

该方案的核心目标是：

1. 在高峰期优先保护数据库  
2. 避免超报名  
3. 避免重复报名  
4. 让用户端感知到清晰、可追踪的报名状态变化  
5. 保证最终报名结果由数据库落库结果决定

---

## 2. 核心设计思路

本方案将报名流程拆成两个阶段：

### 阶段一：接入请求并冻结名额
用户点击报名后，请求先进入后端。  
后端不直接写数据库，而是先通过 Redis 和 Lua 脚本完成：

- 活动有效性基础校验
- 重复报名校验
- 剩余名额判断
- 冻结一个报名名额
- 写入“待确认”状态
- 投递 MQ 消息

此时前端看到的状态是：

**报名确认中**

这一步的重点是：  
先把高频、高并发流量挡在 Redis 层，不让大量请求瞬间冲击数据库。

---

### 阶段二：异步消费并确认最终结果
消息进入 MQ 后，由消费者组异步消费。  
消费者完成以下动作：

- 再次进行业务校验
- 落库报名记录
- 更新数据库报名人数或状态
- 回写 Redis 最终状态
- 释放冻结名额或转为已占用名额
- 通过 WebSocket 向前端推送报名结果

这一步的重点是：  
数据库只处理被 Redis 和 MQ 筛过的一部分真实有效请求，降低瞬时压力。

---

## 3. 用户视角下的报名流程

从用户角度来看，报名过程分为三个状态：

### 3.1 点击报名
用户在活动详情页点击“立即报名”。

此时前端发起报名请求，请求体中至少包含：

- activityId
- userId（由登录态解析）
- requestId 或 traceId（可选）

---

### 3.2 页面进入“报名确认中”
后端请求到达后，先不立即返回最终结果，而是返回：

- 请求已受理
- 当前状态：`PENDING_CONFIRM`

前端拿到这个结果后，将按钮或页面状态更新为：

- 报名确认中
- 正在处理中
- 请稍候

用户此时不会误以为已经成功，而是明确知道系统正在确认。

---

### 3.3 页面收到最终结果
当 MQ 消费者完成数据库落库和状态确认后，后端通过 WebSocket 推送结果到前端。

前端收到推送后，将状态更新为以下之一：

- 报名成功
- 名额已满
- 重复报名
- 报名失败，请重试

这样用户体验会比纯轮询更好，状态变化也更及时。

---

## 4. 报名流程详细设计

## 4.1 报名入口校验
用户请求到达报名接口后，先做基础业务校验：

- 用户是否登录
- 活动是否存在
- 活动是否已发布
- 当前时间是否处于报名时间窗口内
- 活动是否已取消或已结束

这些校验可以先走缓存或数据库，但整体上属于轻量前置校验。

---

## 4.2 Redis 预扣减与冻结名额
如果基础校验通过，进入 Redis 预扣减逻辑。

这里不直接做“正式占用名额”，而是做“**冻结名额**”。

### 为什么要冻结名额
因为此时数据库还没有真正写入报名记录。  
如果 Redis 直接把名额当成最终成功占用，而后续 MQ 消费失败，就会出现 Redis 和数据库不一致的问题。

所以更合理的做法是：

- 先冻结一个名额
- 用户状态标记为“待确认”
- 等数据库落库成功后，再将冻结转为正式占用
- 如果落库失败，再回滚冻结名额

---

### Redis 中建议维护的状态
可以设计如下 Redis 数据：

#### 1. 活动剩余名额
```text
activity:stock:{activityId}
```

表示当前剩余可报名名额。

#### 2. 活动冻结名额
```text
activity:frozen:{activityId}
```

表示当前有多少名额已冻结、待最终确认。

#### 3. 用户报名状态
```text
activity:user:register:{activityId}:{userId}
```

状态可设计为：

- `PENDING`
- `SUCCESS`
- `FAILED`

---

### 预扣减逻辑
建议通过 Lua 脚本把以下动作做成原子操作：

1. 判断活动剩余名额是否大于 0  
2. 判断用户是否已经报名或正在报名中  
3. 如果可以报名，则：
   - 扣减剩余名额
   - 增加冻结名额
   - 把用户状态设为 `PENDING`
4. 返回“受理成功，等待确认”

这样可以同时解决：

- 名额判断
- 防止重复报名
- 冻结名额
- Redis 操作原子性

---

## 4.3 MQ 投递报名事件
Redis 预扣减成功后，不直接写库，而是投递一条报名消息到 MQ。

消息体建议包含：

- activityId
- userId
- requestId / traceId
- createTime
- 幂等标识

这里推荐使用：

- RabbitMQ
- 或 RocketMQ

如果你已经有 RocketMQ 经验，也可以继续沿用 RocketMQ。

---

## 4.4 后端立即返回“报名确认中”
消息成功投递后，报名接口立即返回：

- code: 200
- status: `PENDING_CONFIRM`
- message: 报名确认中，请稍候

这样主链路非常轻，不需要等待数据库写入完成。

---

## 5. 消费者组的异步确认逻辑

消息进入 MQ 后，由消费者组异步处理。

消费者的职责不是简单写库，而是完成一次“最终报名确认”。

---

### 5.1 再次业务校验
虽然前面 Redis 已经过滤过一次，但消费者侧依然建议做必要兜底校验，例如：

- 活动是否仍有效
- 用户是否已存在正式报名记录
- 当前消息是否重复消费

---

### 5.2 数据库落库
消费者在数据库中执行以下动作：

1. 插入报名记录  
2. 更新活动报名统计数据（如已报名人数）  
3. 必要时生成凭证或后续流程数据

这里要保证数据库层也具备防重复能力。

建议：

- 对 `(activity_id, user_id)` 建唯一索引
- 消费端以唯一索引作为幂等兜底

这样即使消息重复消费，也不会真的插入两条报名记录。

---

### 5.3 根据落库结果回写 Redis
数据库落库完成后，再回写 Redis：

#### 如果落库成功
- 用户状态从 `PENDING` 改成 `SUCCESS`
- 冻结名额减少
- 最终占用名额生效

#### 如果落库失败
- 用户状态从 `PENDING` 改成 `FAILED`
- 归还剩余名额
- 释放冻结名额

这里的关键是：

**冻结名额不是最终结果，而是中间状态。**

只有当数据库真正成功后，报名才算最终确认。

---

## 6. WebSocket 推送设计

## 6.1 为什么要用 WebSocket
如果不用 WebSocket，前端通常只能轮询报名状态。  
这样会带来两个问题：

- 轮询频繁，浪费资源
- 用户体验不够实时

因此本方案采用 WebSocket 主动推送。

---

## 6.2 推送时机
当消费者完成数据库落库，并确认最终报名结果后，通过 WebSocket 推送给对应用户。

推送内容可以包括：

- activityId
- userId
- status
- message

例如：

```json
{
  "activityId": 1001,
  "userId": 2001,
  "status": "SUCCESS",
  "message": "报名成功"
}
```

或者：

```json
{
  "activityId": 1001,
  "userId": 2001,
  "status": "FAILED",
  "message": "名额已满"
}
```

---

## 6.3 前端页面更新方式
前端点击报名后，先将页面展示为：

- 报名确认中

建立或复用当前用户的 WebSocket 连接。  
收到服务端推送后，再更新页面：

- 如果状态为 `SUCCESS`，展示“报名成功”
- 如果状态为 `FAILED`，展示具体失败原因
- 并同步刷新报名按钮状态、我的报名记录等界面

---

## 7. 如何保证不超报名

这个方案里，不超报名主要通过两层保证：

### 第一层：Redis 预扣减控制入口
活动剩余名额由 Redis 统一控制。  
只有 Redis 判断还有剩余名额时，才允许进入“待确认”状态。

这样可以把大部分高并发判断挡在 Redis 层。

---

### 第二层：数据库最终落库确认
真正的报名结果，以数据库落库结果为准。

也就是说：

- Redis 只负责入口控制和冻结
- 数据库负责最终确认

如果数据库写入失败，系统会回滚 Redis 中的冻结名额，避免长期不一致。

因此不会出现“前面已经放进去了，后面又写不进去还不回滚”的情况。

---

## 8. 如何防止重复报名

重复报名主要从三层控制：

### 第一层：Redis 用户状态防重
在 Redis 中维护：

```text
activity:user:register:{activityId}:{userId}
```

如果用户已经是：

- `PENDING`
- 或 `SUCCESS`

则直接拒绝再次报名。

---

### 第二层：数据库唯一索引
对 `(activity_id, user_id)` 建唯一索引。  
即使 Redis 层在极端情况下失效，数据库依然可以兜底。

---

### 第三层：MQ 消费幂等
消费者处理消息时，需要保证即使消息重复投递或重复消费，也不会插入重复报名记录。

可以通过：

- 唯一索引
- 消息唯一 ID
- 消费记录表（可选）

来保证消费幂等。

---

## 9. Redis 与数据库不一致问题怎么处理

这个方案必须正视一致性问题。

### 9.1 Redis 冻结成功，但消息发送失败
解决思路：

- 发送 MQ 时增加失败重试
- 或使用本地消息表 / 事务消息
- 至少要有失败补偿机制

对于当前项目，可以先讲：

> 在 Redis 预扣减后，如果 MQ 投递失败，我会通过重试和补偿逻辑恢复冻结名额，避免用户长期停留在“报名确认中”。

---

### 9.2 MQ 消费失败，数据库没写进去
解决思路：

- 消息消费失败自动重试
- 多次失败后记录异常日志并进入补偿任务
- 回滚 Redis 中的冻结名额和用户状态

---

### 9.3 WebSocket 推送失败
解决思路：

- WebSocket 推送失败不影响核心报名结果
- 用户刷新页面时可通过查询接口获取最终状态
- 前端也可以保留降级轮询机制

这点很好讲，因为它体现了：

**推送是增强体验，不是核心事务的一部分。**

---

## 10. 报名状态设计建议

为了让流程更清晰，建议设计用户报名状态如下：

- `PENDING_CONFIRM`：报名确认中
- `SUCCESS`：报名成功
- `FAILED`：报名失败
- `CANCELED`：已取消（可选）

活动名额相关状态可理解为：

- 剩余名额
- 冻结名额
- 已成功占用名额

这样在业务表达上会更清晰，也更适合面试展开。

---

## 11. 技术栈建议

本方案建议的核心技术栈如下：

- **Spring Boot**：承载报名业务逻辑
- **Spring MVC**：提供报名 API
- **MyBatis / MyBatis-Plus**：进行数据库操作
- **MySQL**：存储活动信息、报名记录和最终状态
- **Redis**：预扣减名额、冻结名额、用户状态控制
- **Lua**：保证 Redis 扣减与状态写入原子性
- **RabbitMQ / RocketMQ**：异步削峰、落库确认
- **WebSocket**：向前端主动推送报名最终状态
- **JWT**：解析当前登录用户身份

---

## 12. 接口与事件设计建议

### 12.1 报名接口
```http
POST /api/activities/{activityId}/register
```

返回：

- `PENDING_CONFIRM`
- message: 报名确认中，请稍候

---

### 12.2 查询报名状态接口（兜底）
```http
GET /api/activities/{activityId}/register/status
```

用于前端刷新页面时兜底查询。

---

### 12.3 MQ 事件
事件名称建议为：

- `activity.register.requested`

消费者处理后可在内部形成：

- `activity.register.confirmed`
- `activity.register.failed`

---

### 12.4 WebSocket 推送事件
例如：

- `activity_register_result`

消息体包括：

- activityId
- status
- message

---

## 13. 面试表达建议

如果面试官问“报名模块是怎么设计的”，可以这样回答：

> 我把报名模块设计成了一个异步确认流程，而不是同步插库后立即返回成功。用户点击报名后，后端会先在 Redis 中用 Lua 脚本做原子预扣减：判断活动是否还有剩余名额、判断用户是否已经报名，如果条件满足，就先冻结一个名额，并把用户状态标记为 PENDING，也就是报名确认中。  
>  
> Redis 预扣减成功后，我不会直接写数据库，而是把报名请求投递到 MQ，由消费者异步完成数据库落库。消费者会插入正式报名记录、更新数据库报名状态，并根据最终结果回写 Redis：如果落库成功，就把用户状态改成 SUCCESS；如果失败，就回滚冻结名额，把状态改成 FAILED。  
>  
> 在用户体验上，我没有让前端一直轮询，而是通过 WebSocket 在消费者完成最终确认后主动推送结果给前端，让页面从“报名确认中”实时更新成“报名成功”或“报名失败”。这个方案的核心价值在于：用 Redis 扛住高峰入口流量，用 MQ 做削峰和异步确认，用数据库保证最终一致性，同时通过 WebSocket 改善用户体验。

---

## 14. 模块亮点总结

该报名模块的亮点主要有：

1. **Redis 预扣减 + 冻结名额**
   - 在入口就控制名额
   - 避免高峰请求直接打爆数据库

2. **MQ 异步落库**
   - 将重操作延后处理
   - 削峰填谷，降低数据库瞬时压力

3. **最终一致性设计**
   - Redis 只做冻结和中间状态
   - 报名结果以数据库落库结果为准

4. **WebSocket 状态推送**
   - 前端从“报名确认中”实时切换到最终状态
   - 用户体验更好

5. **多层防重**
   - Redis 防重
   - 数据库唯一索引防重
   - MQ 消费幂等防重

---

## 15. 总结

这个报名模块并不是一个简单的“插一条报名记录”的功能，而是一个围绕高峰流量、名额控制、异步确认和状态回传设计的完整业务链路。

它的设计重点包括：

- 高并发入口控制
- 冻结名额而非直接占用
- 异步落库确认
- 最终一致性
- 实时状态反馈

这套方案既适合校园活动平台的热门报名场景，也非常适合在面试中体现你对缓存、消息队列、异步架构和一致性问题的理解。
